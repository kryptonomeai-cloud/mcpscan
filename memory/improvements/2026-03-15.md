# Improvement Proposal — 2026-03-15

## Problem
**Work loop cycles burn tokens checking blocked tasks repeatedly.** On Mar 14, at least 6 cycles (10:07, 12:22, 13:07, 13:52, 01:10, 01:55) all concluded "all tasks manual or blocked, nothing actionable." Each cycle reads WORKQUEUE.md, checks SSH/Docker, and writes identical conclusions — estimated 30-50k tokens wasted per day on zero-value cycles.

Evidence: Every cycle's output ends with "No automatable work this cycle" or equivalent. The freqtrade zero-trades investigation was valuable the first time but checking "still 0 trades" hourly after diagnosing it as expected behaviour was pure waste.

## Proposed Fix
Add a **blocker-aware skip mechanism** to HEARTBEAT.md or the work loop:
1. Create `memory/work-loop-state.json` tracking `lastActionableTask` timestamp and `blockerHash` (hash of blocked items list)
2. If blockers haven't changed since last check AND no new tasks added, skip the full cycle and just report "no change"
3. Only do full re-evaluation when: (a) >6h since last full check, (b) a new task was added to WORKQUEUE.md, or (c) a blocker was resolved

## Impact
- **Token savings**: ~30-50k tokens/day (6+ redundant cycles eliminated)
- **Signal:noise ratio**: Daily logs become readable — actual events not buried in "nothing happened" entries
- **Cost**: At current model pricing, ~$1-3/day saved on empty cycles

## Risk
- Could miss a blocker resolving if the check interval is too long — mitigated by the 6h max-skip rule
- Needs the work loop to read/write state file — minor complexity

## Implementation
1. Create `memory/work-loop-state.json` with schema: `{ "lastFullCheck": ISO, "blockerHash": string, "taskCount": number }`
2. At start of each work loop cycle, read state file
3. Compare current WORKQUEUE.md task count and blocker list against stored values
4. If unchanged AND <6h since last full check: log one line "No change since HH:MM" and exit
5. Otherwise: run full cycle, update state file
