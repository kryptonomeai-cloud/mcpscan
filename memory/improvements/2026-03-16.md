# Improvement Proposal — 2026-03-16

## Problem
**iCloud Drive hangs break backup cleanup silently.** On Mar 16 at 03:19, the daily backup cron triggered an iCloud Drive hang (bird daemon unresponsive). The 04:00 Paperless export completed but old backup cleanup was skipped because `find`/`ls` hung on the iCloud path. This means stale backups accumulate silently — no alert, no retry, no fallback.

Evidence: Memory entry `2026-03-16T03:19:20` — "bird daemon restarted but FS still hung. Cleanup deferred." And 04:00 entry — "find/ls commands hung; old export cleanup skipped this run."

## Proposed Fix
Add **iCloud health pre-check + timeout wrapper** to backup scripts:
1. Before any iCloud operation, test with `timeout 5 ls ~/Library/Mobile\ Documents/com~apple~CloudDocs/ >/dev/null 2>&1`
2. If test fails: skip iCloud copy, log warning, and write backup to a local fallback path (`~/.openclaw/workspace/backups/`)
3. Add `timeout 30` wrapper around all iCloud file operations (cp, find, rm)
4. Log cleanup failures explicitly so they surface in daily analysis

## Impact
- **Reliability**: Backups never silently skip cleanup — either succeed or fallback + alert
- **Visibility**: iCloud hangs get logged as explicit warnings instead of silent skips
- **Disk**: Prevents unbounded iCloud backup accumulation

## Risk
- Low. Only adds defensive wrappers around existing operations. Fallback path is local disk.
- Could mask persistent iCloud issues if fallback becomes the norm — mitigated by explicit warnings.

## Implementation
1. Edit backup cron scripts to add `icloud_ok()` health check function
2. Wrap iCloud `cp` and `find` in `timeout 30`
3. Add fallback: `~/.openclaw/workspace/backups/` with same retention policy
4. Log "⚠️ iCloud unreachable — used local fallback" when triggered
